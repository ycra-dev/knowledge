# 프로젝트: Astro 기반 지식 블로그 (물리학 · 수학 · 컴퓨터공학)

이 저장소는 **Astro 기반 블로그**입니다. 목표는 “내용은 정확하게, 사이트는 안 깨지게”입니다.

---

## 작업 원칙 (가장 중요)

- **IMPORTANT — Astro 동작 보존**
  - 블로그 수정 요청을 받으면 **Astro의 빌드/라우팅/콘텐츠 컬렉션 동작이 깨지지 않게** 변경한다.
  - 레이아웃/컴포넌트/콘텐츠 컬렉션/MDX/remark·rehype 설정 변경은 특히 조심한다.
  - 확신이 없으면 **먼저 관련 파일(astro.config.*, src/content, layout, integration 설정 등)을 읽고** 변경한다.

- **IMPORTANT — 글 작성은 /CONTRIBUTING_DOCS.md를 최우선 준수**
  - 사용자가 글 작성을 요청하면 **반드시 `CONTRIBUTING_DOCS.md` 규칙(프론트매터/경로/형식/이미지/수식/링크 규칙 등)** 을 따라 작성한다.
  - 규칙을 모르면 먼저 해당 파일을 열어 확인하고, 그 규칙대로만 쓴다.

- **IMPORTANT — 사실 정확성**
  - 수학/물리/컴공 주제로 마크다운 글을 작성할 때는 **사실과 다른 서술이 나오지 않게** 한다.
  - 작성 후 **스스로 1회 재검토**한다:
    - 정의/정리의 가정 조건(“언제 성립하는지”) 누락 여부
    - 기호/단위/차원 일관성
    - 반례 가능성(특히 “항상/반드시” 같은 표현)
    - 알고리즘 복잡도, 경계조건, 예외 케이스
    - 참고문헌/출처/공식 문서 링크(프로젝트 규칙이 허용한다면) 적절성

## 기본 워크플로우 (권장)

1. **탐색(Explore)**: 변경 관련 파일을 먼저 읽고, 프로젝트 구조를 파악한다.
2. **계획(Plan)**: 어떤 파일을 어떻게 바꿀지 짧게 정리한다.
3. **구현(Code)**: 최소 변경으로 적용한다(불필요한 리포맷/대규모 개편 금지).
4. **검증(Check)**: 가능한 범위에서 로컬 검증(빌드/체크)을 수행한다.
5. **마무리(Review)**: 문서/글이면 최종 사실 검토 체크리스트를 한 번 더 돈다.

---

## 프로젝트 컨텍스트 문서(.docs) 운영

- 루트에 **`.docs/` 디렉터리가 없으면 먼저 생성**한다.
- 작업 중 “다음에 또 참고하면 좋은 정보”는 **`.docs/`에 누적 기록**한다.
- 고정 컨텍스트 파일(권장):
  - `.docs/PROJECT_CONTEXT.md`
    - 프로젝트 구조 요약(핵심 디렉터리/콘텐츠 위치)
    - 글 작성 규칙 요약(단, 원문은 `CONTRIBUTING_DOCS.md`가 기준)
    - 자주 건드리는 설정 파일 목록(astro.config.*, src/content/config 등)
    - 자주 쓰는 명령어(실제 package.json 스크립트 기준으로 갱신)

## 코딩/편집 규칙

- **변경 범위는 최소화**: 관련 없는 파일 포맷팅/정렬/리네이밍은 피한다.
- **구성요소 파손 금지**:
  - `src/content`(콘텐츠 컬렉션), `astro.config.*`, MD/MDX 파이프라인 변경은 근거/필요성을 명확히 하고 신중히.
- **명령어는 추측하지 말기**:
  - 실행 전 `package.json`의 `scripts`를 확인해서 그 프로젝트의 명령을 사용한다.
- **글/문서 편집**:
  - 제목/프론트매터/슬러그/카테고리/태그 규칙은 `CONTRIBUTING_DOCS.md`가 최상위.
  - 수식(LaTeX), 코드블록 언어 태그, 링크 스타일도 해당 문서를 따른다.

## 검증(테스트/빌드) 가이드

- 가능한 경우 아래 중 **프로젝트에 실제로 존재하는 것**을 골라 실행한다(반드시 package.json 확인):
  - 빌드: `npm run build` / `pnpm build`
  - 타입/린트/체크: `npm run check` / `npm run lint` / `npm run typecheck`
  - 개발 서버: `npm run dev`
- 변경이 설정/빌드에 영향이 크면 **최소한 build 또는 check 중 하나는 수행**한다.

## 글 품질 체크리스트 (작성 후 1회 수행)

- [ ] 핵심 정의/정리의 **가정(조건)** 을 적었는가?
- [ ] “항상/임의의/모든” 같은 전칭 표현이 **정확히 성립**하는가?
- [ ] 기호가 앞뒤로 바뀌지 않았는가? (예: i/j, n/N, σ/Σ 등)
- [ ] 예제/증명/의사코드가 **본문 주장과 일치**하는가?
- [ ] 알고리즘 복잡도/경계조건/예외가 맞는가?
- [ ] (가능하면) 1~2개 신뢰 가능한 출처(교재/논문/공식 문서)로 교차검증했는가?
- [ ] `CONTRIBUTING_DOCS.md` 형식 요구사항을 모두 만족하는가?
